1これは何を測るものか

このフレームワークは、「AIという確率的なシステムを、実務要件を満たす決定論的なシステムへ組み上げる能力」（Architect Track）を測ります。目的は2つ——①自分の成長領域を特定すること、②社内のスキル分布を可視化して育成・配置の参考にすることです。

測るもの（ARCHITECT TRACK）
  • 要件を厳密に言語化し、AIエージェントを指揮して実装する力
  • コンテキスト・データ・状態の設計力
  • ハルシネーション等の不確実性を定量的に制御する力
  • 本番運用に耐えるインフラ・セキュリティ・コストの設計力
測らないもの（スコープ外）
  • 基盤モデルの研究・事前学習・アーキテクチャ改良
  • Fine-tuning / LoRA等によるモデル適応
  • 手書きコーディングの速度・アルゴリズム競技力
重要 スコープ外の領域に強みを持つ人が低スコアになっても、それは測定対象外というだけで、能力の欠如を意味しません。診断結果を読むときは必ずこの前提を思い出してください。

2証拠の再現性 — 点数上限を決める

この診断は「自分はできます」という主張ではなく、成果物（証拠）を採点します。「発信がうまい人が高く出る」「AI生成記事で実績を盛れる」という落とし穴を防ぐため、証拠を証拠の再現性（A / B / C）に分類し、再現性がそのベクトルの到達度上限を決めます。

A
再現可能
採点時に本人の申告以外の主体（AI・ピア評価者・社内レビュアー）が、手順どおり検証できる成果物。社内非公開でも可（FW 7章）。動くデモURL＋手順、リポジトリURL＋README、Eval数値ファイル＋手順のいずれかがセット。ガイドHTML内にURLと手順を書いてもよい。例: 社内Git、VPN内デモ、共有ドライブ上のEval CSV、HF Spaces
上限 5点（満点まで可）
B
設計文書
「なぜその設計にしたか」の判断・トレードオフが記述された文書。検証用URL・手順・数値記録が無いHTMLガイド単体は原則ここ（B）。SNS投稿・ブログ記事もB。例: 設計思想だけの tech-guide、比較検討記事
上限 3点
C
記述のみ
やったことの記述はあるが、判断理由も再現性もないもの。例: 概要レベルのブログ、箇条書きの実績リスト
上限 2点

つまり、どれだけ経験があっても再現性Aの証拠がないベクトルには4点以上がつきません（Bは上限3点、Cは上限2点）。到達度と再現性は別物です。到達度が高くても再現性Bなら3点止まりになります。

よくある取り違え（採点前に必読）
  • 「第三者」＝インターネット上の不特定一般ではありません。 本人の口だけでない検証主体（採点AI・ピア評価者・社内レビュアー）が手順どおり確認できるか、が基準です。本ツールは社内用のため、社内Git・社内wiki・VPN内デモ・ローカルファイルパスも手順付きならAになり得ます（FW 7章）。
  • HTMLガイドに検証用URL＋手順を書けばAになり得ます。 逆に、設計説明だけで検証用リンク・手順・Eval数値が無いガイドはBです。
  • SNS投稿・ブログ記事は原則B（設計の説明であり、システムの検証先ではない）。
  • 採点AIがセッション内で実際に読めること。 URLを載せる場合、AIがfetchできないときはファイルパス・貼り付け・同セッションで開ける形で提示してください。

証拠の種類と再現性の対応表

提出物の例再現性採点上の意味
社内Git / VPN内デモ / 共有ドライブ ＋ 手順（ガイド内記載可）A社内非公開可。採点AI・ピアが手順どおり検証できること

提出前チェックリスト

診断を始める前に、各ベクトルで次を確認してください。

  • □ 各ベクトルで検証用のURL・ファイル・手順がセットになっているか（社内限定でも可。ガイドHTML内に書いてもよい）
  • □ 採点セッションでAIまたは評価者が実際に内容を確認できる形か（fetch不可のURLだけにしていないか）
  • □ SNS投稿やブログ記事を「検証先の本体」にしていないか（解説として添える分には可）
  • □ 4点以上を期待するベクトルは、再現性Aに加えルーブリック上の到達度も満たしているか

「実力はあるが証拠がない／ガイドだけ」場合、スコアは意図的に低く出ます。最短の改善策は、再現性A相当の成果物を1つ追加することです。

35つの評価ベクトル

スキルを5つの軸（ベクトル）で測ります。各ベクトルは到達度（0〜5）で採点され、下の式で重み付き点に換算して合計100点満点になります。

軸名称重み満点寄与何を見るか
V1Context Engineering
コンテキスト設計力		25%25.0 業務要件・ドメイン知識・タスク状態を、LLMが正確に解釈できる形に翻訳・構造化できる力。曖昧な業務要件を「入力・状態・出力・例外」に分解できるか
V2Agentic System Design
エージェントシステム設計力		25%25.0 AIエージェントを安全に稼働を継続し、実務品質で動くシステムとして設計できる力。セキュリティ（インジェクション対策・権限最小化）と出力品質の制御を含む
V3Knowledge Architecture
知識アーキテクチャ設計力		20%20.0 知識・文書・データをシステムが使える形に整える力。RAG専用ではない（キーワード検索・長文直接投入・「検索基盤を作らない」判断も含む）。複雑なRAGを組んだこと自体は加点されない
V4Evaluation & Observability
評価・観測能力		15%15.0 AIの出力品質を数値で捉え、劣化原因を「プロンプト／データ／モデル／インフラ」に切り分けて改善できる力
V5Production & LLMOps
本番構築・運用能力		15%15.0 AIシステムを本番で動かし続けられる力。インフラ構築だけではない（環境の再現、ローカル／クラウド推論の結合、コスト・データ配置のリスク判断、監視・障害対応の運用を含む）

重み付き点の計算式

重み付き点 = 到達度 × 重み(%) ÷ 5（小数第1位）。合計 = V1〜V5の和（0〜100）。

  • ÷5 の意味: 到達度は0〜5。到達度5のとき、そのベクトルの重み％がそのまま満点寄与になる（例: V1 到達度4 → 4 × 25 ÷ 5 = 20.0）

重みの数値根拠

  • V1・V2（各25%）: Architect Trackの中核。コンテキスト設計とエージェント構造の質に、他軸の成果が従属するため最重視
  • V3（20%）: 実務価値への直結度は高いが、中核2軸の従属側として一段軽くした
  • V4・V5（各15%）: 設計を検証・運用可能にする支援的性格。弱いと合計への打撃は V1/V2 より小さい

重み 25/25/20/15/15 はパイロット前の仮設定で、運用データに基づき改訂され得ます（FW 7章）。各ベクトルの0〜5点の行動基準は正本の3章を参照してください。

5鮮度ルール — 古い実績は効力が落ちる

生成AI分野は主要なパラダイムが半年〜1年で入れ替わるため、実績の作成時期によって証拠の効力を段階的に下げます。

直近12ヶ月
等級どおり満額で評価
A証拠はそのままA（上限5点）として扱われる
12〜24ヶ月
等級を1段階下げる
A→B（上限3点）、B→C（上限2点）に減衰する
24ヶ月超
参考扱い（上限2点）
等級にかかわらず、そのベクトルは最大2点まで

例外

  • 陳腐化の遅い領域（コンテナによる環境分離、ネットワーク・権限設計、評価指標の設計思想など）は、評価者の判断で12〜24ヶ月の実績を満額扱いにできます。適用時は理由が採点シートに記録されます。
  • 逆に、特定モデル・ツールのバージョンに強く依存する実績（特定モデル向けのプロンプト最適化など）は12ヶ月以内でも等級を下げることがあります。
実務上の意味 「昔すごいものを作った」だけではスコアを維持できません。四半期〜半年ごとに再診断し、直近の成果物で証拠を更新し続けることが前提の設計です。

6診断の受け方・使い方

① 個人採点（Claude Codeスキル）

このフォルダ（genai-skill-assessment-app/）を作業ディレクトリとしてClaude Codeの対話モードを起動し、「AIスキル診断を実行して」と入力すると診断が始まります。

1
準備— §2の「提出前チェックリスト」を確認。ガイドHTMLだけでなく、リポジトリ／デモ／Eval数値など一次ソースを揃える
2
基本情報の入力— 対象者名（英数字とハイフン）・評価日・評価者区分（自己評価／ピア／外部）
3
実績の提出— フォルダパス・URL・ファイルで実績を渡す。AIが読み取り
4
証拠引き出しヒアリング（原則）— 見えている穴からプローブを1つずつ。会話だけでは加点しない。不要なら「ヒアリングなしで採点」でスキップ可
5
採点と結果の出力— ベクトル割当・採点のあと outputs/ に個人結果HTML、data/results/ に確定JSON。サマリーを確認し、異議があれば再評価申請

結果の読み方（次の一手と次の壁）

結果HTMLの「次の一手」は、重み付き点が最も低いベクトルに絞った行動設計です。到達度4のベクトルがある場合、その後に「次の壁の見分け（よくある取り違え）」が付きます。これは「4点でできたつもり」になりやすい誤解と、行動基準5の見分けだけを示す任意節で、優先順位は最弱ベクトルのままです（上位○％などの相対表記は出しません）。

証拠引き出しヒアリング（原則実施・分岐型）

原則は採点前に実施します。「ヒアリングして」と依頼する必要はありません。決め打ちの質問リストではなく、見えている穴（シグナル）からプローブを1つずつ選ぶ聞き取りです。会話だけでは加点せず、提出チェックリストと空表の下書きがゴールです。証拠が揃っている場合は確認1ターンで終え、急ぐ場合は「ヒアリングなしで採点」でスキップできます。詳細はスキル定義 .claude/skills/genai-skill-assessment/SKILL.md の「証拠引き出しヒアリング」を参照。

② グループダッシュボード（集計スクリプト）

メンバー各自のJSONを data/results/ に集め、次のコマンドを実行します（Python 3.11+・追加インストール不要・AI呼び出しなし）。

# 既定: data/results/ を集計して outputs/dashboard.html を出力
python3 scripts/generate_dashboard.py

# フォルダ・出力先を指定する場合
python3 scripts/generate_dashboard.py <結果フォルダ> <出力HTMLパス>

社内相対位置・比較レーダーチャート・ヒートマップ・レベル分布が1つのHTMLに出力されます。同一対象者の複数JSONは評価日が最新の1件のみ採用されます。

配布方法

  • 全機能（推奨）: genai-skill-assessment-app/ フォルダごとコピーする（前提: Python 3.11+ と Claude Code本体のみ）
  • スキルのみ: .claude/skills/genai-skill-assessment/ を利用先プロジェクトの .claude/skills/ にコピー
  • 集計スクリプトのみ: scripts/generate_dashboard.py 単体をコピー（標準ライブラリのみで動作）
セキュリティ制約（最重要） 個人採点は必ず人間が立ち会う対話セッションで実行してください。claude -p 等の無人・自動のヘッドレス呼び出しは社内セキュリティポリシーにより使用禁止です。集計スクリプトはAI呼び出しを一切含まず、オフラインで動作します。

7よくある誤解と運用原則

スコアが低い＝エンジニアとして能力が低い、ということ？
違います。測っているのはArchitect Trackの、しかも「証拠が残っている」能力だけです。L0は「評価可能な実績が未蓄積」という意味であり、能力の否定ではありません。成果物を残す習慣がない人は実力より低く出ます（既知の限界）。
「社内Top X%」のような順位はわからないの？
現時点では出せません。相対表記は、社内で10名以上の採点データが揃い、評価者間の一致が確認できるまで凍結されています。いま読み取れるのは絶対スコアとレベル帯のみです。「次の壁の見分け」も順位ではなく、よくある取り違え（概念の壁）を示します。
到達度4なのに「まだ足りない」と書かれるのは減点？
減点ではありません。4はすでに高い到達です。「次の壁の見分け」は、5に見える成果（例: スキーマ運用・役割分担）と、本当の5（例: 状態受け渡し規約・本番の継続運用記録）を取り違えないための気づきです。今週の行動は、結果HTMLの主たる「次の一手」を優先してください。
この結果は人事評価（昇給・査定）に使われる？
使われません。結果は育成・配置の参考情報であり、処遇査定には直接連動させないことが運用原則です。また、結果は本人に全項目開示され、追加証拠の提出により再評価を申請できます。
「第三者」とはインターネット上の見知らぬ人のこと？
違います。 本人の申告だけでなく、採点AI・ピア評価者・社内レビュアーが手順どおり検証できるか、という意味です。本ツールは社内用なので、社内Git・社内wiki・VPN内デモ・ローカルファイルも手順付きなら再現性Aになり得ます（FW 7章「社内非公開の成果物を等級Aとして認める運用」）。外部公開は必須ではありません。
HTMLガイドの中にURLと手順を書けばAになる？
なり得ます。 ガイド内に「デモURL＋確認手順」「リポジトリ＋再現手順」「Eval CSVの場所＋再計測方法」がセットで書いてあり、採点セッションでAIが内容を確認できるならAです。逆に、設計説明だけで検証先・手順・数値が無いガイドはB（上限3点）のままです。
URLを載せれば再現性A（再現可能）になる？
URLの種類と手順が重要です。 SNS投稿・ブログ記事は原則Bです。Aになるのは、リポジトリ・デモ・Eval数値ファイルなど検証可能な一次ソースに、再現・確認手順がセットになっている場合です（社内限定でも可）。
HTML技術資料（spec-tech-guide）だけ渡すと点数が下がる？
中身次第です。 設計説明だけなら再現性B（上限3点）。ガイド内に検証用URL・手順・Eval数値が書いてあればAになり得ます。ガイドに何も書かずファイルだけ渡す場合も、検証情報が無ければBです。
スコアを上げる最短の方法は？
最も低いベクトルについて「再現性A（評価者が手順どおり検証できる成果物）」を1つ作ることです。リポジトリ＋手順、デモURL＋手順、数値記録つきEvalでも可。再現性Aは上限5点まで開き、到達度が足りれば4点以上も可能です。
自己評価だけで完結していい？
自己評価から始めて構いませんが、会話・自己申告のみを根拠にした採点は禁止されています（成果物の棚卸しが先）。組織で使う場合は、評価者2名の独立採点で基準を校正してから展開する設計です。
