## 1. スコープ宣言

### 対象とする能力（Architect Track）

AIという確率的なシステムを、実務要件を満たす決定論的なシステムへ組み上げる能力。具体的には：

- 要件・仕様を厳密に言語化し、AIコーディングエージェントを指揮して実装する力
- コンテキスト・データ・状態の設計力
- 不確実性（ハルシネーション・精度のブレ）を定量的に制御する力
- 本番運用に耐えるインフラ・セキュリティ・コストの設計力

### 対象外（明示的スコープ外）

以下は本フレームワークでは**測定しない**。これらに強みを持つ人材が低スコアになっても、それはスコープ外であることを意味するだけで、能力の欠如を意味しない。

- 基盤モデルの研究・事前学習・アーキテクチャ改良
- Fine-tuning / LoRA等によるモデル適応（将来「Model Track」として別途定義する可能性あり）
- 手書きコーディングの速度・アルゴリズム実装競技力

## 2. 証拠等級制（採点の前提ルール）

実績ベース採点の最大の落とし穴は「発信力とスキルの混同」および「AI生成記事による見かけの実績」である。これを防ぐため、証拠を3等級に分類し、等級が各ベクトルの**上限点**を決める。

| 等級 | 定義 | 例 | そのベクトルの上限点 |
|------|------|-----|---------------------|
| **A（再現可能）** | 第三者が再現・検証できる成果物。コード＋データ＋手順、動くシステム、定量的な計測結果を含む | GitHubリポジトリ、動作するデモ、設定ファイル込みの構築手順、Eval結果の数値記録 | 5点 |
| **B（設計判断の文書）** | 「なぜその設計にしたか」の判断・トレードオフが記述された文書 | アーキテクチャ設計書、比較検討記事、HTML技術資料 | 3点 |
| **C（記述のみ）** | やったことの記述はあるが、判断理由も再現性もないもの | 概要レベルのブログ、箇条書きの実績リスト | 2点 |

### 付帯ルール

- **鮮度ルール（3段階）**: 生成AI分野の主要パラダイムは半年〜1年で入れ替わるため、実績の作成時期で証拠力を減衰させる。
  | 実績の時期 | 扱い |
  |-----------|------|
  | 直近12ヶ月 | 等級どおり満額で評価 |
  | 12〜24ヶ月 | 等級を1段階下げる（A→B、B→C） |
  | 24ヶ月超 | 参考扱い。そのベクトルの上限2点 |

  **鮮度例外**: 陳腐化の遅いスキル領域（コンテナによる環境分離、ネットワーク・権限設計、評価指標の設計思想など、LLMの世代交代の影響を受けにくいもの）に限り、評価者の判断で12〜24ヶ月の実績を満額扱いにできる。例外を適用した場合は採点シートに理由を記載すること。逆に、モデル・ツールの特定バージョンに強く依存する実績（特定モデル向けのプロンプト最適化など）は12ヶ月以内でも等級を下げてよい。

## 3. 評価ベクトル（5軸・100点満点）

### 重みの根拠

V1・V2を最重視（各25%）するのは、Architect Trackの中核が「コンテキストとエージェント構造の設計」であり、他の4軸の成果がこの2軸の質に従属するため。V3（20%）は実務価値への直結度、V4・V5（各15%）は「設計を検証・運用可能にする」支援的性格から相対的に軽くした。この重みはパイロット後に再検討する。

| ベクトル | 重み | 到達度5のときの満点寄与 | 数値根拠（要約） |
|---------|------|----------------------|------------------|
| V1 Context Engineering | 25% | 25.0 | 中核。他軸が従属 |
| V2 Agentic System Design | 25% | 25.0 | 中核。他軸が従属 |
| V3 Knowledge Architecture | 20% | 20.0 | 実務直結だが従属側 |
| V4 Evaluation & Observability | 15% | 15.0 | 検証の支援的性格 |
| V5 Production & LLMOps | 15% | 15.0 | 運用の支援的性格 |
| **合計** | **100%** | **100.0** | |

### 重み付き点の計算式

- **重み付き点** = 到達度 × 重み(%) ÷ 5（小数第1位まで）
- **合計点** = V1〜V5の重み付き点の和（0〜100）
- **÷5 の意味**: 到達度は0〜5スケール。到達度5のとき、そのベクトルの重み％がそのまま満点寄与になるように正規化する（例: 到達度4・重み25% → 4 × 25 ÷ 5 = 20.0）
- V4/V5=15% のため、到達度が同じでも合計への寄与上限は V1/V2 より小さい

### V1: Context Engineering — コンテキスト設計力（25%）

**定義**: 業務要件・ドメイン知識・タスク状態を、LLMが最も正確に解釈できる形に翻訳・構造化・配置できる能力。**ビジネス要件をAIシステム要件に翻訳できる力を含む。**

| 点 | 行動基準（ツール非依存） |
|----|------------------------|
| 0 | プロンプトの試行錯誤のみ。設計の記録がない |
| 1 | 基本的なプロンプト技法（例示提示・段階的推論の誘導）を意図的に使い分けている |
| 2 | 長大な入力に対する分割・要約・優先順位付けの戦略を設計し、文書化している |
| 3 | コンテキストの構造的劣化（中間情報の欠落等）を予見した階層的コンテキスト設計を実装している |
| 4 | ドメイン固有のコンテキストスキーマ（構造化された仕様記述形式）を定義し、再現可能な形で運用している |
| 5 | 複数エージェント間の状態受け渡し規約を設計・実装し、要件の翻訳から状態設計まで一貫した方法論として文書化している |

**必須の採点観点**: 曖昧な業務要件を「入力・状態・出力・例外」に分解した実績があるか。「作らない／単純な解で済ませる」判断をした記録があるか。

---

### V2: Agentic System Design — エージェントシステム設計力（25%）

**定義**: 自律型AIエージェント（単一・複数）を、安全に稼働を継続し、実務品質で動くシステムとして設計できる能力。**セキュリティと出力品質（UI/UX含む）の制御を含む。**

| 点 | 行動基準（ツール非依存） |
|----|------------------------|
| 0 | チャットUIの利用のみ |
| 1 | ノーコード／ローコード基盤で基本的なワークフローを構築できる |
| 2 | ツール呼び出し（外部API・検索・計算）を組み込んだエージェントを構築している |
| 3 | 複数ツールを連携させたエージェントを構築し、失敗時のフォールバックを設計している |
| 4 | 複数エージェントの役割分担（計画・実行・検証等）を設計し、暴走防止（ループ上限・状態機械・人間の承認点）を実装している |
| 5 | 本番相当のエージェントシステムを、安全機構・権限設計・出力品質制御を含む一貫した設計文書とともに構築・運用している |

**必須の採点観点**（3点以上を付ける場合、以下を確認する）:
- **セキュリティ**: プロンプトインジェクション対策、エージェントに与える権限の最小化を考慮しているか
- **出力品質・UI制御**: 生成させたUIや文書が実務品質か。デザイン原則・出力規約を事前に明文化してエージェントに与えているか（AI生成物特有の冗長・装飾過多を排除できているか）

---

### V3: Retrieval & Knowledge Architecture — 知識アーキテクチャ設計力（20%）

**定義**: ドメイン知識・文書・データを、システムが使える形に整え、課題に応じた参照方法を設計できる能力。**RAG（ベクトル検索）に限定しない**（キーワード・ハイブリッド・再ランキング・長文コンテキスト直接投入、「検索基盤を作らない」判断を含む）。重要なのは要否と適正レベルの判断であり、複雑な検索基盤を構築したこと自体は加点しない。**

| 点 | 行動基準（ツール非依存） |
|----|------------------------|
| 0 | 知識参照の仕組みを構築した実績がない |
| 1 | 文書を分割してベクトル検索する基本構成を構築している |
| 2 | メタデータ設計・フィルタリングにより検索対象を制御している |
| 3 | 課題特性に応じて検索方式（キーワード・ベクトル・ハイブリッド・再ランキング・長文コンテキスト直接投入）を**比較検討し、根拠をもって選定**している |
| 4 | 複雑な文書（図表・数式・多階層構造）に対するパース・インデックス戦略を設計し、精度を定量比較している |
| 5 | ドメイン全体の知識構造（階層・関係性・鮮度）を設計し、検索精度の改善を数値で実証している。「検索基盤を作らない」判断を含む最適化ができている |

**注意**: 3点の「長文コンテキスト直接投入で十分と判断した」ケースは、根拠が示されていれば複雑なRAG構築と同等に評価する。

---

### V4: Evaluation & Observability — 評価・観測能力（15%）

**定義**: AI（確率的に振る舞うシステム）の出力品質を数値で捉え、劣化の原因を「プロンプト／検索データ／モデル／インフラ」に切り分けて改善できる能力。

| 点 | 行動基準（ツール非依存） |
|----|------------------------|
| 0 | 目視確認のみ |
| 1 | 良否判定の基準を決めて手動評価している |
| 2 | テストケース集（想定入力と期待出力）を作成・維持している |
| 3 | 忠実性・関連性等の指標を定義し、計測している |
| 4 | 自動評価パイプラインを構築し、変更のたびに回帰評価している |
| 5 | 評価結果に基づいて原因を峻別し、改善前後の数値比較を伴う改善サイクルを文書化・公開している |

---

### V5: Production & LLMOps — 本番構築・運用能力（15%）

**定義**: AIシステムを本番で動かし続けられる能力。**インフラ構築だけではない**（環境の分離・再現、推論基盤との結合、コストとデータのリスク判断、監視・障害対応・コスト管理の継続運用を含む）。

| 点 | 行動基準（ツール非依存） |
|----|------------------------|
| 0 | ブラウザ上のサービス利用のみ |
| 1 | API呼び出しを含むスクリプトをローカルで実行できる |
| 2 | コンテナ技術で依存関係を分離した開発環境を構築・文書化している |
| 3 | ローカル推論環境とデータベースを構築し、アプリケーションと結合している |
| 4 | モデル選定（商用API vs OSS）・量子化・インフラ構成を、**コスト試算とデータ漏洩リスク評価を根拠に**選定している |
| 5 | 監視・ログ・障害対応・コスト管理を含む運用を継続し、その記録を文書化している |

**必須の採点観点**（4点以上）: 月次コストの試算根拠、データの置き場所（ローカル／クラウド）の選定理由が文書に含まれているか。

> **ツール例（参考・非拘束）**: コンテナ=Docker等 / ローカル推論=Ollama, vLLM等 / ベクトルDB=Qdrant, Chroma等 / 評価=Ragas, LangSmith, Phoenix等 / ワークフロー=Dify, LangGraph等。ツールは陳腐化するため、採点はあくまで行動基準で行う。

## 6. 運用設計

### 6.1 自己位置把握の手順（個人向け）

1. **実績棚卸し**: 直近24ヶ月の成果物（公開・社内）をベクトル別にリストアップし、各証拠に等級（A/B/C）と鮮度区分（12ヶ月以内 / 12〜24ヶ月 / 24ヶ月超）を付ける
2. **自己採点**: 証拠等級の上限ルールを厳守して採点する
3. **ギャップ特定**: 「実力はあるが等級A証拠がない」ベクトルを特定する。→ そのベクトルの再現可能な成果物を1つ作ることが最短の改善策（最も低いベクトルを次四半期の焦点にする）
4. **次の壁の見分け（任意）**: 到達度4のベクトルがある場合、「よくある取り違え」と行動基準5の見分けを確認する（例: スキーマ運用≠エージェント間の状態受け渡し規約）。パーセンタイルや称賛は使わない。行動の優先は手順3のまま
5. **再測定**: 四半期ごとに再採点し、スコアの推移を記録する

### 6.2 社内可視化の手順（組織向け）

1. **パイロット**: 2〜3名を対象に、評価者2名が独立採点する
2. **キャリブレーション**: ベクトル単位で2名の採点が2点以上乖離した場合、その行動基準を書き直して再採点する
3. **本展開**: 基準が安定してから対象を広げる。結果はレーダーチャート（5軸）とチームヒートマップで可視化する
4. **育成への接続**: 最も低いベクトル1つに対して「等級A証拠を1つ作る」を次四半期の育成目標にする

### 6.3 HR運用原則（必須）

- 評価結果は**本人に全項目開示**する（採点理由・根拠資料を含む）
- 本人は**再評価を申請できる**（追加証拠の提出により）
- 結果は育成・配置の**参考情報**であり、昇給・査定に直接連動させない
- スコープ外宣言（1章）を評価対象者に事前に周知する

### 6.4 フレームワーク自体の保守

- **半年ごとに見直し**、版番号を更新する（この分野の陳腐化速度に対応）
- 見直し対象: ベクトル定義の妥当性、重み、ツール例、レベル閾値
- 変更はすべて冒頭の「方針決定記録」形式で追記する

## 7. 既知の限界（利用者への注意）

本フレームワークには以下の限界があり、結果の解釈時に考慮すること。

1. **相対位置は未校正**: 世界・業界内での位置（Top X%）はまだ主張できない。現時点で測れるのは「本ルーブリック上の絶対スコア」と「社内での相対位置（パイロット後）」のみ
2. **発信バイアスの残存**: 証拠等級制で緩和しているが、成果物を残す習慣がない人は依然として過小評価される。社内非公開の成果物を等級Aとして認める運用でこれを補うこと
3. **スコープの偏り**: Architect Track に特化しており、モデル開発層・伝統的ソフトウェア工学の深い実装力は測定対象外
4. **重みの暫定性**: 25/25/20/15/15 はパイロット前の仮設定であり、運用データに基づき改訂される
5. **業界標準との未対応**: 既存の資格体系（JDLA、クラウドベンダー認定等）や公開ベンチマークとの対応付けは未実施。外部調査の上、次版で対応表を追加予定
